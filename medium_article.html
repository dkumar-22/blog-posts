Reducing Latency in Asynchronous Microservices: From 3 Seconds to 100ms

How we transformed a blocking synchronous architecture into a resilient async system while maintaining data consistency and retry capabilities

The Problem

When building distributed systems, one of the most common challenges is dealing with synchronous calls that block your entire request pipeline. Our middleware service—built on Apache Camel—was orchestrating message processing across multiple downstream services. The flow looked simple enough:

  1. Receive message from queue
  2. Call downstream service synchronously
  3. Wait for response (up to 3 seconds)
  4. Acknowledge or retry based on response

The downstream service performed several operations: database writes, cloud storage movements (like moving files between S3 buckets), and potentially sending notifications. These operations were critical and needed to complete successfully, but they were also time-consuming.

The core issue: Our middleware was waiting 3 seconds for each message, creating a bottleneck. With thousands of messages per day and concurrent processing enabled, this synchronous blocking was killing our throughput.

The Challenge: Cross-Region Complexity

Adding to the complexity, our downstream service could be deployed in a different region than the middleware. This meant:

  • Higher network latency for API calls
  • Potential for regional failures
  • Callbacks might face network issues when returning results

We needed a solution that would:

  1. Make the middleware respond quickly (reduce blocking time)
  2. Ensure the downstream service still completed all operations reliably
  3. Maintain idempotency (prevent duplicate processing on retries)
  4. Preserve the middleware's role as the orchestrator for retries

The Solution: Async Processing with Smart Orchestration

Step 1: Make the Downstream Service Return Immediately

The first breakthrough was realizing the downstream service didn't need to complete everything before responding. Instead:

// Before
receive_request() {
    move_files_in_storage()  // 2.5 seconds
    update_database()        // 400ms
    send_notifications()     // 100ms
    return success           // Total: 3+ seconds
}

// After
receive_request() {
    check_idempotency()      // 50ms
    save_to_db(PENDING)      // 50ms
    return 202 ACCEPTED      // Total: 100ms
    
    // Process asynchronously
    background_worker {
        move_files_in_storage()
        update_database()
        send_callback_to_middleware()
    }
}

Step 2: Implement Idempotency Checks

Since we were moving to an async model where retries were inevitable, idempotency became critical. Every request now includes a unique identifier (like an email ID or transaction ID).

Before processing, the downstream service checks:

idempotency_check(request_id) {
    status = db.get_status(request_id)
    
    if status == COMPLETED:
        return 200 OK  // Already processed
    
    if status == PROCESSING:
        return 200 OK  // Currently being processed
    
    if status == null or FAILED:
        db.insert(request_id, PENDING)
        return 202 ACCEPTED  // New request, will process
}

This ensures that even if the middleware retries due to network issues or timeouts, we never process the same request twice.

Step 3: Queue-Based Orchestration

The middleware manages three queues:

  • pending: Messages waiting to be processed
  • in-progress: Messages currently being processed by downstream service
  • dlq (dead letter queue): Messages that failed after maximum retry attempts

Flow:

1. Consume from 'pending' queue
2. Call downstream service
3. If 202 received → Move to 'in-progress' queue
4. Wait for callback from downstream service
5. Success callback → ACK (remove from queue)
6. Failure callback → Move back to 'pending' (retry) or 'dlq' (max retries exceeded)

Step 4: Callback Mechanism with Retry

The downstream service sends callbacks to inform the middleware of the final result. To make this reliable:

  • Track callback status in database (PENDING, DELIVERED, FAILED)
  • Retry failed callbacks with exponential backoff (30s, 1min, 2min, 5min)
  • After max retries, alert operations team for manual intervention

Step 5: Handling Stuck Messages

One edge case: what if a message gets stuck in 'in-progress' because the downstream service crashed or the callback was lost?

Solution: A timeout monitor runs every 5 minutes and moves messages stuck in 'in-progress' for more than 10 minutes back to 'pending'. The idempotency check ensures these won't be processed twice—if they were already completed, the service returns 200 on retry.

Handling Data Consistency

One critical requirement was maintaining transactional consistency between storage operations and database updates. Our approach:

  1. State Machine Pattern: Track status transitions (PENDING → PROCESSING → COMPLETED/FAILED)
  2. Database First: Always update the database before deleting old storage files
  3. Compensation Logic: If storage succeeds but database fails, mark as FAILED and retry the entire operation

The key insight: Storage operations (like S3 moves) are idempotent. Copying the same file twice produces the same result. So retrying the entire operation is safe.

The Challenge of Selective Retry with RabbitMQ

One interesting problem we encountered: With concurrent processing (5 worker threads), we might have 10 messages being processed simultaneously. When callbacks arrive saying "message 4 and 7 failed," how do we selectively retry just those messages?

The RabbitMQ limitation: You cannot selectively consume specific messages from the middle of a queue. RabbitMQ is FIFO (First In, First Out).

Our solution: Fan-out pattern—write each message to both:

  • RabbitMQ queue: For ordering and distribution
  • Cache/Database: For storing the original message content

When a failure callback arrives with a correlation ID:

  1. Look up the original message in cache using correlation ID
  2. Retrieve the full message content
  3. Increment retry counter
  4. Re-publish to 'pending' queue with original content

Note: If you're using Kafka instead of RabbitMQ, this is simpler—Kafka messages aren't deleted on consumption and support offset-based replay naturally.

Results

After implementing this architecture:

  • Middleware latency: Reduced from 3 seconds to <100ms (97% improvement)
  • Throughput: Increased by 30x (can process 30 messages in the time it took to process 1)
  • Reliability: Zero duplicate processing due to idempotency
  • Resilience: Automatic retry for transient failures, DLQ for permanent failures
  • Cross-region support: Works reliably even with higher network latency

Key Takeaways

  1. Async is not just about threads: It's about changing your API contract. Return 202 Accepted and process in the background.
  2. Idempotency is non-negotiable: In async systems with retries, idempotency checks prevent data corruption.
  3. Middleware should orchestrate, not wait: Let downstream services manage their internal processing. Middleware should manage flow and retries.
  4. Queue + Storage pattern: Use message queues for ordering/distribution, use cache/DB for message persistence during async processing.
  5. State machines are your friend: Explicit state tracking (PENDING → PROCESSING → COMPLETED) makes systems debuggable and predictable.
  6. Plan for failure: Callbacks can fail. Network can fail. Services can crash. Design for it upfront with retries, timeouts, and DLQs.

Considerations and Trade-offs

Eventual Consistency: The middleware gets a 202 immediately, but the actual processing completes later (usually within seconds). This is fine for most use cases, but if you need strong consistency, you'll need a different approach.

Operational Complexity: Async systems are harder to debug. You need good observability—metrics, logs, and traces that connect requests with their async processing and callbacks.

Message Storage: Storing messages in cache/DB during processing adds infrastructure. We used Redis with TTL to automatically clean up after processing completes.

When NOT to Use This Pattern

This architecture works well for:

  • High-volume message processing
  • Operations that take >1 second
  • Systems where eventual consistency is acceptable
  • Cross-region deployments with variable latency

Consider synchronous processing if:

  • Operations complete in <100ms
  • You need immediate confirmation of success/failure
  • Your system is simple and doesn't justify the async complexity

Final Thoughts

Moving from synchronous to asynchronous processing isn't just a technical change—it's a shift in how you think about distributed systems. Instead of "call and wait," you move to "trigger and track." Instead of hoping networks are fast, you design for them to be slow and unreliable.

The result? Systems that are faster, more resilient, and can scale horizontally without hitting blocking bottlenecks.

If you're dealing with similar latency issues in your microservices architecture, start by identifying your slowest synchronous calls and ask: "Does this really need to be synchronous?" Often, the answer is no—and that's your opportunity to optimize.
